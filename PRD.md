# PRD: 단말 지향 Mutation 기반 SIP 프로토콜 퍼저

## 1. 문서 목적
본 문서는 상용 단말(UE)을 대상으로 하는 Mutation 기반 SIP 프로토콜 퍼저의 **목표, 범위, 기능 요구사항, 비기능 요구사항, 단계별 산출물, 완료 기준**을 정의한다. 이 문서는 프로젝트의 전체 설계 범위와 완료 기준을 명확히 하고, 이후 문서(`1-단말기준-SIP-메시지-전체분류-및-정의.md`)와 코드 설계의 기준점으로 사용한다.

## 2. 배경 및 문제 정의
SIP는 VoLTE, IMS, IP Phone, Softphone 등 다양한 환경에서 세션 생성·변경·종료를 담당하는 텍스트 기반 신호 프로토콜이다. SIP는 헤더 조합이 다양하고 메시지 간 상태 의존성이 강하므로, 단순 랜덤 입력만으로는 의미 있는 테스트 케이스를 만들기 어렵다.

특히 상용 단말은 소스코드 접근이 어려운 경우가 많아 정적 분석보다 **블랙박스 기반 퍼징**이 현실적이다. 따라서 본 프로젝트는 RFC/IANA 기반의 정상 SIP 메시지를 생성한 뒤 이를 변조하여 단말 방향으로 전송하고, 단말의 크래시·ANR·재부팅·비정상 응답을 탐지하는 퍼저를 목표로 한다.

## 3. 제품 목표
1. 단말(UE) 기준으로 **수신 가능한 SIP 메시지 전체 목록과 각 메시지의 의미/전제조건**을 완전하게 정의한다.
2. 분류된 전체 SIP Request/Response 유형에 대해 **정상 SIP 메시지 생성기(Generator)** 를 구현한다.
3. 전체 메시지 유형을 대상으로 **재현 가능한 Mutation 엔진(Mutator)** 을 구현한다.
4. 변조된 메시지를 단말로 전달하고 반응을 처리하는 **Sender/Reactor** 를 구현한다.
5. 단말의 크래시·ANR·재부팅·SIP 스택 이상 징후를 탐지하는 **Oracle** 을 구현한다.
6. 전체 퍼징 캠페인을 실행·중지·기록하는 **CLI 기반 Controller** 를 구현한다.
7. 모든 테스트 결과를 재현·분석 가능한 형태로 저장하고, 졸업프로젝트 수준의 완결된 산출물로 정리한다.

## 4. 프로젝트 원칙 및 경계
1. RFC/IANA 기준으로 확인되는 단말 기준 수신 가능 SIP 메시지 종류는 **문서와 스키마에서 모두 포함**한다.
2. 구현 또한 전체 범위를 목표로 하며, 개발 순서는 공통 기반 기능 → 핵심 시나리오 → 확장 시나리오 순으로 진행할 수 있다.
3. 본 프로젝트의 중심 대상은 단말(UE) 지향 퍼징이며, 서버/프록시/IMS 구성요소는 단말 테스트를 성립시키는 환경 요소로 다룬다.
4. 본 프로젝트는 연구 및 검증 목적이며, 취약점 악용 자동화나 무기화는 목표로 하지 않는다.
5. 단일 트랜잭션뿐 아니라 dialog, 재협상, 이벤트 구독/통지, 확장 응답 등 복합 상태 시나리오까지 포함하는 것을 목표로 한다.

## 5. 대상 환경 및 사용자
### 5.1 대상 시스템
- SIP를 처리하는 단말(UE)
- 우선 대상: VoLTE/IMS 계열 단말 또는 이에 준하는 SIP 처리 단말
- 테스트 환경: 연구용 사설 네트워크 또는 통제된 실험 환경

### 5.2 예상 사용자
- 모바일/프로토콜 보안 연구자
- SIP/IMS 동작을 분석하는 개발자
- 단말의 SIP 처리 안정성을 검증하려는 테스트 엔지니어

## 6. 범위 정의
### 6.1 메시지 방향 범위
본 프로젝트는 **단말 기준 수신 메시지(incoming to UE)** 를 기준으로 설계한다.

- Incoming Request: 네트워크/프록시/상대 단말이 UE에 보내는 요청
- Incoming Response: UE가 보낸 요청에 대해 네트워크가 UE로 반환하는 응답

### 6.2 전체 분류와 전체 구현 목표
Phase 1에서는 RFC/IANA 기준으로 **단말 기준 수신 가능한 전체 메시지 목록** 을 정리한다.
이후 단계에서는 문서와 스키마뿐 아니라 실제 구현에서도 이 전체 범위를 지원하는 것을 목표로 한다.
개발 순서를 관리하기 위한 내부 우선순위 표시는 둘 수 있지만, **최종 프로젝트 범위에서 제외되는 메시지 종류는 두지 않는다.**

### 6.3 전체 구현 대상 메시지
#### A. Incoming Request 전체 대상 (IANA 등록 14개)
- ACK
- BYE
- CANCEL
- INFO
- INVITE
- MESSAGE
- NOTIFY
- OPTIONS
- PRACK
- PUBLISH
- REFER
- REGISTER
- SUBSCRIBE
- UPDATE

#### B. Incoming Response 전체 대상 (IANA 등록 75개)
- `1xx Informational`
- `2xx Success`
- `3xx Redirection`
- `4xx Client Error`
- `5xx Server Error`
- `6xx Global Failure`
- 개별 Response Code와 세부 의미는 `1-단말기준-SIP-메시지-전체분류-및-정의.md` 에서 코드별로 관리한다.

### 6.4 구현 순서 원칙
- 공통 파싱/스키마/상태 관리 기반을 먼저 구축한다.
- 이후 핵심 호출 시나리오와 확장 시나리오를 순차적으로 넓혀 간다.
- 단, 이 순서는 개발 효율을 위한 것이며 **최종 산출물은 전체 메시지 범위를 포함**해야 한다.

## 8. 기능 요구사항(Functional Requirements)
### FR-1. SIP 메시지 분류
시스템은 단말 기준 수신 가능한 SIP 메시지 **전체 목록** 을 구조적으로 분류하고 정의해야 한다.

최소 포함 정보:
- 메시지 유형: Request / Response
- Method 또는 Status Code
- Direction: incoming to UE
- UE 역할: UAS / UAC
- 적용 전제조건(예: 특정 요청 이후에만 수신 가능)
- 구현 상태 또는 검증 상태 표시
- 근거 RFC/IANA 참조
- 간단한 메시지 의미/용도 설명

### FR-2. Generator
시스템은 분류된 **모든 SIP Request/Response 유형** 에 대해 정상적인 SIP 메시지를 생성할 수 있어야 한다.

최소 요구사항:
- 요청 메서드별 기본 템플릿 제공
- 응답 코드별 기본 템플릿 또는 생성 규칙 제공
- 필수 헤더 자동 채움
- 대화/트랜잭션 문맥에 필요한 상태 유지
- Pydantic 스키마를 통한 입력 검증

### FR-3. Mutator
시스템은 정상 Request/Response 메시지를 기반으로 다양한 변조를 수행할 수 있어야 한다.

최소 요구사항:
- 헤더 삭제/중복/순서 변경
- 헤더 값 변조
- 상태 코드/메서드/Reason-Phrase 관련 변조
- 본문/길이 관련 변조
- 상태 의존 필드 변조(Call-ID, CSeq 등)
- 랜덤 시드 기반 재현 가능성 보장

### FR-4. Sender/Reactor
시스템은 생성/변조된 메시지를 전송하고 단말 또는 네트워크의 반응을 기록해야 한다.

최소 요구사항:
- 테스트 케이스 단위 전송
- 응답/타임아웃/연결 실패 기록
- 요청-응답 상관관계 추적
- 캠페인 실행 중 로그 저장

### FR-5. Oracle
시스템은 퍼징 중 단말의 비정상 상태를 탐지해야 한다.

최소 요구사항:
- 프로세스 크래시
- ANR
- 재부팅
- SIP 스택 비정상 종료 또는 응답 상실
- 이상 징후 발생 시 관련 입력과 환경 정보 기록

### FR-6. Controller
시스템은 CLI 기반으로 전체 퍼징 워크플로우를 제어해야 한다.

최소 요구사항:
- 메시지 분류 확인
- 정상 메시지 생성
- 변조 실행
- 단일 테스트 전송
- 캠페인 단위 실행
- 결과 저장 경로/시드/대상 설정 관리

### FR-7. 결과 기록 및 재현성
시스템은 모든 테스트 케이스를 재현 가능한 형태로 저장해야 한다.

최소 포함 정보:
- test case ID
- seed
- 원본 메시지 / 변조 메시지
- 전송 시각
- 대상 단말 정보
- 관측된 응답 또는 이상 징후

## 9. 비기능 요구사항(Non-Functional Requirements)
1. **단일 책임 원칙(SRP)** 을 준수한다.
2. 프로젝트 환경 관리는 **uv** 를 사용한다.
3. 코드 품질 관리는 다음 기준을 따른다.
   - `ruff format`
   - `ruff check`
   - `ty check`
4. CLI 프레임워크는 **Typer** 를 사용한다.
5. 데이터 스키마는 **Pydantic** 으로 정의한다.
6. 핵심 로직은 모듈 단위로 테스트 가능해야 한다.
7. 동일 seed에 대해 가능한 한 동일한 테스트 케이스를 재생성할 수 있어야 한다.
8. 로그는 사람이 읽기 쉬우면서도 후처리 가능한 구조를 가져야 한다.

## 10. 단계별 진행 순서 및 완료 기준
### Phase 1. SIP 메시지 분류
목표:
- 단말 기준 수신 가능한 SIP 메시지의 **모든 종류** 를 분류하고 정의한다.
- 각 메시지의 의미, 전제조건, 단말 관점 역할, 관련 RFC/IANA 근거를 구조화한다.

산출물:
- `1-단말기준-SIP-메시지-전체분류-및-정의.md`
- 전체 메시지 분류를 담는 Pydantic 기반 message catalog/schema

완료 기준:
- 단말 기준 수신 가능한 모든 Request Method와 모든 Response Code가 누락 없이 정리되어 있다.
- 각 항목에 대해 Request/Response, UE 역할, 의미, 전제조건, RFC/IANA 근거가 명시되어 있다.
- 이후 Generator가 사람이 수동 해석하지 않고 사용할 수 있는 구조화된 데이터 모델이 정의되어 있다.

### Phase 2. Generator 작성
목표:
- 분류된 전체 SIP Request/Response 유형에 대한 정상 메시지 생성 기능을 구현한다.

완료 기준:
- 14개 Request Method 각각에 대해 최소 1개 이상의 정상 템플릿 또는 생성 규칙을 제공한다.
- 등록된 모든 Response Code에 대해 기본 생성 규칙과 코드-문구 매핑을 제공한다.
- 필수 헤더 누락 없이 메시지를 생성한다.
- 스키마 검증을 통과한다.

### Phase 3. Mutator 작성
목표:
- 전체 메시지 유형을 대상으로 의미 있는 변조를 수행한다.

완료 기준:
- Request와 Response 모두에 대해 구조 변조, 값 변조, 상태 의존 변조를 각각 최소 1종 이상 지원한다.
- seed를 저장하고 동일 seed로 동일 테스트 케이스를 재현할 수 있다.

### Phase 4. Sender/Reactor 작성
목표:
- 테스트 케이스를 실제로 전송하고 반응을 수집한다.

완료 기준:
- 단일 테스트 케이스 전송이 가능하다.
- 응답/타임아웃/실패가 구분되어 기록된다.
- 요청과 응답의 상관관계를 추적할 수 있다.

### Phase 5. Oracle 작성
목표:
- 단말 비정상 징후를 자동으로 탐지한다.

완료 기준:
- 최소 1개 이상의 단말 이상 징후(예: 크래시 또는 ANR)를 자동 탐지한다.
- 이상 징후 발생 시 관련 테스트 케이스와 로그를 연결해 저장한다.

### Phase 6. Controller 작성
목표:
- 전체 퍼징 흐름을 CLI에서 일관되게 실행한다.

완료 기준:
- 사용자가 CLI 명령으로 분류 확인, 생성, 변조, 전송, 캠페인 실행을 수행할 수 있다.
- 결과 산출물이 지정 디렉터리에 구조적으로 저장된다.

## 11. 성공 기준(Success Criteria)
다음 조건을 만족하면 프로젝트 목표를 달성한 것으로 본다.

1. 단말 기준 수신 가능한 전체 SIP 메시지 분류가 문서와 스키마 양쪽에 반영되어 있다.
2. 14개 Request Method와 등록된 전체 Response Code에 대한 생성 규칙이 구현되어 있다.
3. Request와 Response 모두에 대해 변조 → 전송 → 반응 수집 → 이상 징후 기록의 흐름이 end-to-end로 동작한다.
4. 동일 seed 기반 재현이 가능하다.
5. 최소 1개의 실제 단말 테스트 세션에서 전체 파이프라인 결과 로그를 안정적으로 수집할 수 있다.

## 12. 오픈 이슈
다음 항목은 후속 문서 또는 구현 단계에서 구체화가 필요하다.

1. 초기 타깃 단말 종류와 OS 버전
2. 실험 네트워크 구성 방식(IMS 유사 환경, 사설 SIP 환경 등)
3. 인증/등록 상태를 퍼저가 전제할지 여부
4. 전송 프로토콜 우선순위(UDP/TCP 등)
5. Oracle에서 사용할 관측 채널(ADB, logcat, 네트워크 타임아웃 등)
