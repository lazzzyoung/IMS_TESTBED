# SIP 응답 패킷 예시

> 이 문서는 `scripts/generate_packet_docs.py` 로 생성된다.
> 이 문서는 각 SIP 응답을 **실제 SIP 텍스트 패킷 형태**로 이해하기 위한 설명 문서이다.
> 예시 패킷은 대표 예시이며, 응답 코드별 핵심 헤더만 우선 보여준다.

## 공통 응답 골격

```text
SIP/2.0 200 OK
Via: SIP/2.0/UDP proxy.example.com;branch=z9hG4bK-generic
From: "UE" <sip:ue@example.com>;tag=ue-tag
To: "Network" <sip:network@ims.example.com>;tag=net-tag
Call-ID: generic-response@example.com
CSeq: 1 INVITE
Content-Length: 0
```

공통 필수 헤더:
- `Via`
- `From`
- `To`
- `Call-ID`
- `CSeq`

## 100 Trying

- 설명: Provisional response indicating request processing has started.
- 대표 상황: UE sent a request and the next hop has begun processing it.
- 관련 메서드: ACK, BYE, CANCEL, INFO, INVITE, MESSAGE, NOTIFY, OPTIONS, PRACK, PUBLISH, REFER, REGISTER, SUBSCRIBE, UPDATE
- RFC: RFC3261

## 180 Ringing

- 설명: Indicates the callee is alerting.
- 대표 상황: UE receives 180 after sending INVITE for an outbound call.
- 관련 메서드: INVITE
- RFC: RFC3261

### 대표 SIP 패킷 예시

```text
SIP/2.0 180 Ringing
Via: SIP/2.0/UDP proxy.example.com;branch=z9hG4bK-180-invite
From: "UE" <sip:ue@example.com>;tag=ue-tag
To: "Network" <sip:network@ims.example.com>;tag=net-tag
Call-ID: 180-invite@example.com
CSeq: 1 INVITE
Require: 100rel
RSeq: 1
Content-Length: 0
```

### 필수 헤더
- `Via`
- `From`
- `To`
- `Call-ID`
- `CSeq`

### 대표 선택/조건부 헤더
- `Require`
- `Rseq`

### 조건부 규칙
- `Rseq`: Include RSeq when the provisional response is sent reliably with 100rel. — RSeq is not mandatory for ordinary provisional responses, only for reliable ones.
- `Contact`: For non-100 provisional responses to INVITE that establish an early dialog, include Contact so the remote target is known.
- `Record-Route`: If the INVITE request contained Record-Route, copy it into the dialog-establishing provisional response.

## 200 OK

- 설명: Generic success response for SIP requests.
- 대표 상황: UE request completed successfully.
- 관련 메서드: ACK, BYE, CANCEL, INFO, INVITE, MESSAGE, NOTIFY, OPTIONS, PRACK, PUBLISH, REFER, REGISTER, SUBSCRIBE, UPDATE
- RFC: RFC3261

### 대표 SIP 패킷 예시

```text
SIP/2.0 200 OK
Via: SIP/2.0/UDP proxy.example.com;branch=z9hG4bK-200-invite
From: "UE" <sip:ue@example.com>;tag=ue-tag
To: "Network" <sip:network@ims.example.com>;tag=net-tag
Call-ID: 200-invite@example.com
CSeq: 1 INVITE
Contact: <sip:network@ims.example.com>
Content-Type: application/sdp
Content-Length: 98

v=0
o=- 0 0 IN IP4 ims.example.com
s=-
c=IN IP4 198.51.100.20
t=0 0
m=audio 50002 RTP/AVP 0
```

### 필수 헤더
- `Via`
- `From`
- `To`
- `Call-ID`
- `CSeq`

### 대표 선택/조건부 헤더
- `Contact`
- `Content-Type`

### 조건부 규칙
- `Contact`: Required for 2xx to INVITE and other dialog-establishing usages where a remote target must be learned.
- `Expires`: In a 200-class response to SUBSCRIBE, Expires indicates the actual subscription duration granted by the notifier.
- `Contact`: A 2xx response to MESSAGE MUST NOT include Contact because MESSAGE does not establish a dialog.
- `Body`: A 2xx response to MESSAGE MUST NOT include a message body.

## 202 Accepted (Deprecated)

- 설명: Accepted but not yet fully completed; still valid for MESSAGE asynchronous handling, deprecated for SUBSCRIBE, and obsoleted for new REFER usage.
- 대표 상황: Asynchronous MESSAGE processing, with legacy interoperability from older SUBSCRIBE/REFER deployments still possible.
- 관련 메서드: MESSAGE

## 204 No Notification

- 설명: Successful SUBSCRIBE processing without sending a follow-up NOTIFY.
- 대표 상황: In-dialog SUBSCRIBE refresh is accepted and the notifier suppresses an immediate NOTIFY.
