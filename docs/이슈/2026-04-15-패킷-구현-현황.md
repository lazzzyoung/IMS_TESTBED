# 2026-04-15 · MT 패킷 메서드별 구현 현황 및 부족한 부분

## 목적

real-ue-direct 캠페인에서 `--methods` 로 지정 가능한 SIP 메서드들이 **어디까지 구현되어 있고 어디가 비어 있는지**를 한 곳에 정리한다. 이번 문서는 작업 계획이 아니라 **현 상태 스냅샷**이다 — 어떤 항목을 보강할지는 별도 RFC에서 결정한다.

## TL;DR — 메서드별 매트릭스

|---|---|---|---|---|---|---|---|---|---|

## 패킷 생성 경로

real-ue-direct 캠페인의 hot path 는 메서드에 따라 두 갈래로 나뉜다 (`campaign/core.py:954-984`).

- **INVITE**: `generator/templates/mt_invite_a31.sip.tmpl` 로드 → `render_mt_invite()` 슬롯 치환 (`real_ue_mt_template.py`)
- **그 외**: `generator/mt_packet.py::build_mt_packet()` 로 헤더 조립

소프트폰 모드 (`--target-host ...`) 는 메서드 무관하게 `SIPGenerator` (Pydantic 모델 + catalog) 경로만 사용한다.

## 항목별 현황

### 1. CLI · Catalog 정의 vs 실제 사용
- `sip/common.py::SIPMethod` 에 14 메서드 모두 정의, `sip/catalog.py:73` 에서 14 개 검증.
- `campaign/contracts.py:100` 의 기본 methods 는 `("INVITE",)` — 사용자가 명시하지 않으면 INVITE 만 퍼즈.
- CLI 는 14 메서드 모두 받지만 `real_ue_mt_template.py` 모듈 docstring 부터 "MT-INVITE 전용" 이라 명시.

### 2. Body 생성
- `mt_packet.py:65-93` 의 `_build_body()`:
  - INVITE → AMR-WB SDP 동적 생성
  - MESSAGE → seed 기반 deterministic `text/plain` 기본 body
  - INFO → 기본 `Info-Package: dtmf` + `application/dtmf-relay`
  - PUBLISH → 기본 `Event: presence` + `application/pidf+xml`
  - NOTIFY → 기본 `Event: presence` + `Subscription-State` + `application/pidf+xml`
  - NOTIFY (`event_package="refer"`) → `message/sipfrag;version=2.0`
  - UPDATE → 기본 `application/sdp`
  - OPTIONS / SUBSCRIBE / REFER / PRACK / REGISTER / ACK / BYE / CANCEL → 빈 body
- `event_package` / `info_package` 를 받는 wire-text builder는 추가되었지만, 현재 campaign/CLI hot path 에서는 해당 값을 주입하는 플래그/설정 경로가 없다. 따라서 실사용에서는 기본값(`presence`, `dtmf`)만 사용된다.
- Pydantic 쪽 `BodyFactory` 의 의미(plain text / dtmf / pidf / sipfrag / sdp)를 일부 wire-text path 에 재사용·반영했지만, 완전히 동일한 생성 모델은 아니다.

### 3. Teardown / Idle 대기 (INVITE-only)
`campaign/core.py:1074-1095` 의 `is_invite` 가드 3 곳:
- `collect_all_responses=is_invite` (line 1077) — 비-INVITE 는 final 응답 1 개만 수집 (1xx 무시). SIP 의미상 정상.
- `_teardown_invite()` (line 1084) — INVITE 만 CANCEL 재시도. Stateless 메서드는 teardown 불필요하므로 의도적.
- `wait_for_idle()` (line 1092) — INVITE 만 device idle 복귀 대기. **MESSAGE/OPTIONS 는 송신 후 device 상태 확인 없이 다음 케이스로 진행**. 캠페인 속도에는 유리하지만 device 가 활성 상태로 남았을 때 검출 수단이 없음.

### 4. Mutator
`mutator/core.py` 는 메서드 독립 — `EditableSIPMessage` 구조만 본다. 모든 layer/strategy 가 어떤 메서드에도 적용 가능.
- `_SUPPORTED_MODEL_TARGETS` (call_id, cseq, request_uri, tag 등) 는 모든 메서드가 가지는 헤더라 OK.
- MESSAGE 는 기본 body 가 seed 기반으로 바뀌어, body 자체도 재현 가능한 입력 다양성을 가진다.

### 5. Oracle
`oracle/core.py::SocketOracle.evaluate` 는 `OracleContext.method` 를 받지만 사용하지 않는다. 모든 메서드의 200 OK 가 동일하게 `normal`. SIP 표준 의미상 맞으나, 메서드별 특화 판정 (예: MESSAGE 의 202 Accepted, OPTIONS 의 Allow 헤더 검증) 여지는 열려 있음.

### 6. Sender
`sender/core.py::SIPSenderReactor` 와 `sender/real_ue.py::RealUEDirectResolver` 모두 메서드 무관. real-ue-direct 모드의 MSISDN→IP/포트 해석은 INVITE 외 메서드도 그대로 동작.

### 7. 테스트 커버리지
- INVITE: `tests/campaign/test_*.py` 다수 (template, dialog integration, crash analysis, teardown).
- OPTIONS: `tests/campaign/test_dialog_integration.py:89` 1 건 (stateless path) + `tests/generator/test_mt_packet.py` unit 1건.
- MESSAGE / SUBSCRIBE / INFO / PUBLISH / NOTIFY / UPDATE: `tests/generator/test_mt_packet.py` 에 direct unit coverage 추가.
- 여전히 REFER / PRACK / REGISTER / ACK / BYE / CANCEL 의 MT wire-text direct unit 테스트는 없음.

### 8. 실기기 검증
- A31 ring 성공은 INVITE 한정 (`docs/A31_REAL_UE_GUIDE.md`, `CLAUDE.md`).
- MESSAGE / OPTIONS 등은 형식상 송신 가능하나 **A31 / iPhone 에서 동작 확인 기록 없음**.

### 9. 템플릿 중복
`mt_invite_a31.sip.tmpl` 과 `mt_invite_3gpp.sip.tmpl` 은 **바이트 단위 동일** (`diff` 결과 0). `--mt-invite-template a31` 과 `3gpp` 가 사실상 같은 결과를 만든다.

## 부족한 부분 요약

### 우선순위 높음 (실사용에 영향)
1. **MESSAGE / OPTIONS 의 idle 대기 부재** — INVITE 외 메서드에서도 device 가 비정상 상태로 남는 경우를 어떻게 감지할지 미정. 현재는 다음 케이스가 그 위에 누적됨.
2. **실기기 검증 공백** — A31 / iPhone 에서 INVITE 외 메서드가 실제 어떻게 처리되는지 베이스라인 측정 없음.
3. **event/info package 지정 경로 부재** — wire-text builder는 `event_package` / `info_package` 를 지원하지만, 현재 campaign/CLI에서 사용자 입력으로 전달하는 플래그나 config 필드가 없다.

### 우선순위 중간 (정확성 / 다양성)
4. **REFER / PRACK / REGISTER body·헤더 시나리오 빈약** — `NOTIFY refer` 는 보강됐지만 REFER request 자체의 `Refer-To` 등은 MT wire-text path 에 반영되지 않았다.
5. **테스트 공백 잔존** — REFER / PRACK / REGISTER / ACK / BYE / CANCEL 의 MT wire-text direct 테스트는 여전히 없음.
6. **Oracle 의 메서드 인지 부재** — `OracleContext.method` 가 전달되지만 미사용. 메서드별 정상/이상 응답 코드 매트릭스를 가지면 false-positive 줄일 여지.

### 우선순위 낮음 (정리)
7. **템플릿 중복** — `mt_invite_a31.sip.tmpl` 과 `mt_invite_3gpp.sip.tmpl` 동일. 하나로 통합하거나 차별화.
8. **문서화 갭** — `CLAUDE.md` / `docs/USAGE.md` 에 "INVITE 외 메서드는 best-effort, 실기기 검증 없음" 명시 필요.
