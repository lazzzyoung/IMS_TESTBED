## 세션 3: INVITE teardown 안정화

### 3-1. Reliable CANCEL with retry (`561e599`)

**배경**: 세션 1에서 추가한 best-effort CANCEL이 fire-and-forget이라 패킷 유실
시 단말이 ringing 상태에 걸림. 장기 퍼징에서 이후 케이스가 전부 오염될 수 있다.

**문제 분석**:
1. CANCEL 보내고 응답 안 기다림
2. 통화 상태 확인 없음 (Oracle이 call state를 체크하지 않음)
3. cooldown은 있지만 cleanup 아님
4. 연속 INVITE 누적 가능

**변경**: `_teardown_invite` 메서드 신규:

| 상황 | 동작 |
|------|------|
| 응답 없음 (timeout) | CANCEL 스킵 — 서버가 INVITE를 안 받아 transaction state 없음 |
| final만 (486 등) | CANCEL 스킵 — transaction 이미 종료 |
| provisional 수신 (100/180) | CANCEL 송신 + 200 OK 또는 487 응답 확인 |
| CANCEL 응답 없음/실패 | 최대 3회 재시도 (0.5초 간격) |
| 3회 모두 실패 | observer event 기록 + 2초 추가 cooldown |

**핵심**:
- RFC 3261 §9.1 준수: provisional 응답 받은 후에만 CANCEL
- `collect_all_responses=True`로 응답 확인
- 재시도로 패킷 유실 대비
- 최종 실패 시 fallback cooldown으로 단말 자체 timer 만료 대기

**파일**: `campaign/core.py` (+120줄), `tests/campaign/test_core.py` (+178줄, 6 tests)

---

### 3-2. ADB 기반 통화 상태 확인 (`6da4013`)

**배경**: CANCEL teardown이 성공해도 단말이 실제로 IDLE 상태로 돌아왔는지 확인할
수 없었다. 단말이 RINGING/OFFHOOK 상태에서 다음 INVITE를 보내면 결과가 오염된다.

**변경**: `CallStateChecker` 클래스 신규 (`adb/call_state.py`):
- `get_call_state()` — `adb shell dumpsys telephony.registry`에서 `mCallState` 파싱
  - `0` = IDLE, `1` = RINGING, `2` = OFFHOOK, 실패 시 UNKNOWN
- `is_idle()` — IDLE 여부 반환
- `wait_for_idle()` — IDLE이 아니면 0.5초 간격 폴링, 최대 10초 대기
  - 이미 IDLE → 즉시 리턴
  - UNKNOWN → 스킵 (ADB 연결 문제일 때 퍼징 중단 방지)
  - RINGING/OFFHOOK → 폴링 후 IDLE 전환 확인
  - 타임아웃 → 경고 이벤트 기록 + 강제 진행

**캠페인 통합** (`campaign/core.py`):
- `CampaignExecutor.__init__`: ADB 활성 + MT template 경로일 때 자동 생성
- `_execute_mt_template_case`: CANCEL teardown 직후 `wait_for_idle()` 호출

**실행 흐름**:
```
INVITE 송신 → 응답 수신 → CANCEL teardown (retry 3회)
  → [call state 확인: IDLE 대기] → Oracle 판정 → 다음 케이스
```

**파일**: `adb/call_state.py` (신규, 110줄), `campaign/core.py` (+15줄),
`tests/adb/test_call_state.py` (신규, 11 tests)

### INVITE 장기 퍼징 안정성 이슈 해결 요약

세션 3에서 식별한 4개 이슈와 해결 현황:

| # | 이슈 | 상태 | 해결 방법 |
|---|------|------|-----------|
| 1 | CANCEL이 best-effort | **해결** | `561e599` — reliable CANCEL retry (3회, 200/487 응답 확인) |
| 2 | 통화 상태 확인 없음 | **해결** | `6da4013` — CallStateChecker (dumpsys telephony.registry 폴링) |
| 3 | cooldown ≠ cleanup | **해결** | cleanup이 `_execute_mt_template_case` 내부에서 완료됨. cooldown은 rate limiter로만 동작 |
| 4 | 연속 INVITE 누적 | **해결** | #1 + #2 조합으로 방지: CANCEL 확인 → IDLE 대기 → 다음 INVITE |

최종 케이스 간 흐름:
```
케이스 N:  INVITE → 응답 → CANCEL (retry 3회) → wait_for_idle (ADB IDLE 확인)
                                                        ↓ (IDLE 확인됨)
main loop:                                         cooldown sleep (rate limit)
                                                        ↓
케이스 N+1: INVITE → ...
```

## 세션 4: 장기 퍼징 케이스 생성

### 4-1. 라운드 순환 CaseGenerator + 무한 모드 (`8033439`)

**배경**: `CaseGenerator`가 method×layer×strategy 유니크 조합을 한 번만 생성하고
끝나서, `--max-cases 1000`을 지정해도 조합이 4개면 4개만 실행되었다.
장기 퍼징에서 같은 전략을 다른 시드로 반복 실행해야 다양한 변이 결과를 얻을 수 있다.

**변경**:

| 항목 | Before | After |
|------|--------|-------|
| 조합 고갈 시 | 즉시 종료 | 다음 라운드로 순환 (시드 증가) |
| identity baseline | 매번 생성 | Round 0에서만 1회 |
| `max_cases=0` | 불가 (ge=1) | 무한 모드 (Ctrl+C까지) |
| 대시보드 표시 | `[5/1000]` | `[5/∞]` (무한 모드) |

**라운드 순환 동작**:
```
Round 0: [identity, (wire,default), (byte,default)]  seed=0,1,2
Round 1: [(wire,default), (byte,default)]             seed=3,4
Round 2: [(wire,default), (byte,default)]             seed=5,6
...
```

- 조합 목록은 1회 생성 (라운드 내 중복 제거)
- `seed = seed_start + case_id`로 자연스럽게 다른 변이 생성
- identity는 오라클 baseline이므로 첫 라운드에서만 실행
- `recurring_combos`가 비어있으면 (identity만 있었으면) Round 0 후 종료

**파일**: `campaign/core.py` (CaseGenerator), `campaign/contracts.py` (max_cases ge=0),
`campaign/dashboard.py` (∞ 표시), `tests/campaign/test_core.py` (+3 tests),
`tests/campaign/test_contracts.py` (테스트 수정)

## 세션 5: IPsec 우회 분석 + 범용 3GPP 패킷 + ADB/pcap 강화

### 5-1. Capstone Dumpipe IPsec 우회 분석 문서

**배경**: Capstone(졸업프로젝트 전 버전) fuzzer의 IPsec 우회 방식을 코드 기반으로 분석.

**발견**:
- Capstone은 `#!define WITH_IPSEC` 활성화 + `ipsec_preferred_ealg "null"` 설정
- Dumpipe 모듈이 P-CSCF를 통째로 건너뛰고 UPF 경유 UE 5060 포트로 직접 전송
- Source IP 스푸핑/xfrm 회피 없음 — 소프트웨어 UE 전용
- 우리 프로젝트는 P-CSCF netns에서 Source IP 스푸핑 + xfrm policy 미매치(port 15100)

**파일**: `docs/이슈/Capstone-Dumpipe-IPsec-우회-분석.md` (신규)

### 5-2. P-CSCF IP 불필요 실험 검증

**배경**: "P-CSCF IP(172.22.0.21)로 스푸핑해야 UE가 받는가?" 실험.

**실험**:
```
호스트 br-volte IP(172.22.0.1):15100 → UE(10.20.20.2):9800
결과: 200 OK (User-Agent: TTA-VoLTE/3.0 SM-A315N)
```

**발견**: A31은 transport source IP를 검증하지 않음. P-CSCF IP 스푸핑 없이
호스트에서 직접 전송해도 UE가 수신+응답. docker exec가 필수가 아닐 수 있음.

### 5-3. `--mt` 플래그 — 범용 3GPP 패킷 모드

**배경**: `--mt-invite-template a31 --preserve-via --preserve-contact --ipsec-mode null`을
매번 지정해야 했음. 또한 INVITE 전용이라 MESSAGE/OPTIONS 등은 3GPP 포맷 미지원.

**변경**:
- `--mt` 플래그 추가 (campaign + send request 양쪽)
- `--mt` 설정 시 자동 적용: `mt_invite_template=3gpp`, `preserve_via=True`,
  `preserve_contact=True`, `ipsec_mode=null`
- `build_mt_packet()` 신규 — 모든 SIP 메서드에 3GPP 공통 헤더 생성
  - 공통: Request-URI(alias), Via, P-Access-Network-Info, P-Asserted-Identity,
    P-Charging-Vector, P-Visited-Network-ID, Contact(feature tags)
  - INVITE: AMR-WB SDP, Record-Route, Session-Expires
  - MESSAGE: text/plain body
  - OPTIONS/BYE/CANCEL 등: body 없음
- campaign `_execute_mt_template_case`에서 INVITE는 기존 파일 템플릿,
  나머지 메서드는 `build_mt_packet` 사용
- `CaseGenerator`에서 INVITE identity 강제 주입 제거 — `--methods MESSAGE`로
  MESSAGE만 보낼 수 있게 됨

**사용법**:
```bash
# Before
uv run fuzzer campaign run --mode real-ue-direct --target-msisdn 222222 \
  --impi 001010000123511 --mt-invite-template a31 --ipsec-mode null \
  --preserve-via --preserve-contact --methods INVITE --max-cases 1

# After
uv run fuzzer campaign run --mode real-ue-direct --target-msisdn 222222 \
  --methods MESSAGE --mt --max-cases 1
```

**파일**: `generator/mt_packet.py` (신규), `campaign/contracts.py`, `campaign/cli.py`,
`campaign/core.py`, `sender/cli.py`
