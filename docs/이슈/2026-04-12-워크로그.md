## 2. null encryption 모드 수정

### 문제
`--ipsec-mode null` 모드가 호스트에서 source IP 172.22.0.21로 spoofing 시도 → `Network is unreachable` 에러.
`ip_nonlocal_bind=1` 설정해도 라우팅 문제로 실패.

### 원인
- `campaign/contracts.py`: null 모드에서 `source_ip="172.22.0.21"`, `bind_container=None` 설정
- `campaign/core.py:787`: `_execute_mt_template_case`에서도 동일하게 덮어씀

### 수정
두 곳 모두 null 모드에서 bypass와 동일하게 pcscf 컨테이너 내부 송신으로 변경:
```python
# contracts.py & core.py
if config.ipsec_mode in ("null", "bypass"):
    source_ip = None
    bind_container = "pcscf"
```

### 검증
```
--ipsec-mode null → normal (180) ✅
--ipsec-mode bypass → normal (180) ✅
```

## 3. UE IP 동적 resolve

### 문제
`--target-host` 없이 `--target-msisdn 222222`만 지정하면, 정적 매핑 테이블(`_DEFAULT_MSISDN_TO_IP`)에서 잘못된 IP(10.20.20.9)를 반환. 실제 UE IP는 EPC가 동적으로 할당하므로 매번 변경됨 (10.20.20.2 → 10.20.20.3 → 10.20.20.5 → ...).

### 수정
- `campaign/contracts.py`: 정적 `resolve_ue_ip_from_msisdn()` 호출 제거 → `target_host=None` 허용
- `sender/contracts.py`: 동일하게 정적 resolve 제거
- `campaign/core.py`: `_execute_mt_template_case`에서 `RealUEDirectResolver.resolve(target, impi=...)` 호출로 동적 resolve
- `sender/real_ue.py`: 두 가지 새 resolver 메서드 추가
  - `_lookup_via_pcscf_options_ping()`: P-CSCF OPTIONS keepalive 로그에서 `OPTIONS to sip:<impi>@<ip>:<port>` 파싱
  - `_lookup_via_xfrm_state()`: xfrm state에서 UE IP 파싱 (fallback)

### 검증
```
--target-msisdn 222222 (--target-host 없이) → 자동 resolve → normal (180) ✅
```

## 4. IMPI 동적 resolve

### 문제
`--impi 001010000123512`를 매번 지정해야 함. MSISDN 222222와 IMPI 001010000123512는 문자열 관계가 없어서 매칭 불가.

### 수정
- `UEContact` / `ResolvedRealUETarget`에 `impi: str | None` 필드 추가
- `_lookup_via_pcscf_options_ping`: IMPI를 함께 반환, 매칭 없으면 최신 항목을 fallback으로 반환 (단일 UE 환경)
- `campaign/core.py`: resolve 결과에서 IMPI를 가져와 template slot에 사용
- `campaign/contracts.py`: `VMF_IMPI` 환경변수 fallback, impi 필수 검증 제거

### 검증
```
--target-msisdn 222222 (--impi 없이) → IMPI 자동 추출 → normal (180) ✅
```

## 7. 패킷 캡처 및 IPsec 분석

### IMS REGISTER + IPsec SA 수립 흐름 확인
- 1차 REGISTER (평문 UDP 5060): `Security-Client` 헤더로 UE의 IPsec 파라미터 제안
- 401 Unauthorized: `Security-Server` 헤더로 P-CSCF 파라미터 응답 + `WWW-Authenticate` (AKA challenge)
- 2차 REGISTER (ESP 암호화): `Security-Verify` + AKA response → IPsec SA 위에서 전송
- 200 OK: `P-Associated-URI`에서 MSISDN↔IMPI 매핑 확인

### 암호화 알고리즘
- 무결성: HMAC-SHA-1-96
- 암호화: AES-CBC
- 프로토콜: ESP transport mode

### eNB 구간 (GTP 터널) 캡처 확인

```bash
sudo tcpdump -i br-volte host 172.22.0.50 and udp port 2152 -A
```

| 패킷 | IP Protocol | 내용 |
|------|------------|------|
| 1차 REGISTER | UDP(17) 평문 | `REGISTER sip:...` 텍스트 보임 |
| 401 응답 | UDP(17) 평문 | `Security-Server` 텍스트 보임 |
| 2차 REGISTER~ | ESP(50) 암호화 | 깨진 바이트만 보임 |

### fuzzer INVITE 캡처 확인

```bash
# 캡처
sudo tcpdump -i br-volte host 172.22.0.50 and udp port 2152 -A
# 동시에 INVITE 전송
uv run fuzzer campaign run --mode real-ue-direct --target-msisdn 222222 ...
```

결과: fuzzer의 INVITE와 180 Ringing 모두 **평문으로 확인**.

```
INVITE sip:001010000123512@10.20.20.14:9000 ... ← 평문
SIP/2.0 180 Ringing ...                         ← 평문
```

### 평문 통과 원리
- fuzzer는 sport **15100**으로 송신
- xfrm selector는 sport **5100**(또는 재등록 시마다 변경되는 포트)만 매칭
- 15100은 selector에 안 걸림 → ESP 미적용 → 평문 UDP 통과

### 2개 구간 개별 pcap 캡처
```bash
# IPsec 구간 (eNB ↔ SGW/UPF)
sudo tcpdump -i br-volte host 172.22.0.50 and udp port 2152 -w /tmp/pcap_ipsec.pcap

# P-CSCF 이후 구간 (P-CSCF ↔ I-CSCF)
sudo tcpdump -i br-volte host 172.22.0.21 and tcp port 5060 -w /tmp/pcap_core.pcap
```

### Phase 2: Wire/Byte 변이

#### ADB false positive 문제
- `RILJ.*Error` 패턴이 A31의 정상 백그라운드 로그에 항상 존재
- 모든 케이스가 `stack_failure`로 판정되는 false positive 발생
- `--circuit-breaker 0`으로 비활성화하여 우회

#### 케이스 생성기 제한 (핵심 이슈)

MT template 경로의 `CaseGenerator`가 `(method × layer × strategy)` 카트시안 곱의 **고유 조합당 1개만** 생성하고 종료한다.

예: `--methods INVITE --layer wire,byte --strategy identity,default` 지정 시:
```
INVITE × wire   × identity  → seed=0 (1개)
INVITE × wire   × default   → seed=1 (1개)
INVITE × byte   × identity  → seed=2 (1개)  
INVITE × byte   × default   → seed=3 (1개)
                               총 4케이스로 종료
```

`--max-cases 50`을 지정해도 4개만 실행된다. 일반(softphone) 퍼징 경로에서는 같은 조합을 seed를 바꿔가며 반복하지만, MT template 경로에서는 이 로직이 빠져있다.

**현재 우회법**: bash 루프로 `--seed-start`를 바꿔가며 반복 실행
```bash
for i in $(seq 0 10 90); do
  uv run fuzzer campaign run ... --seed-start $i --max-cases 2 ...
done
```

**근본 수정 필요**: `CaseGenerator`에서 MT template 모드일 때도 `max_cases`까지 seed를 증가시키며 동일 조합을 반복 생성하도록 수정

#### 세션 정리 문제
- INVITE 후 BYE/CANCEL 미전송 → UE가 이전 세션 유지 → 다음 INVITE에 486 Busy 또는 timeout
- `_execute_mt_template_case`에 best-effort CANCEL 전송 로직 추가

#### 10라운드 결과 (seed 0~90, 각 2케이스)

| Verdict | 수 | 비율 |
|---------|-----|------|
| timeout | 14 | 70% |
| normal | 6 | 30% |

응답 코드 분포: 180(Ringing), 486(Busy), 603(Decline), 606(Not Acceptable)

crash/stack_failure 0건 — UE IMS 스택 안정적.

## 10. Wave 1 퍼징 결과 (seed 0~40, 79 cases)

CANCEL 로직 추가 후 루프 실행 (`--seed-start` 0~40, 각 2케이스, wire/default + wire/identity)

### Verdict 분포

| Verdict | 수 | 비율 |
|---------|-----|------|
| timeout | 46 | 58% |
| unknown | 26 | 33% |
| normal | 7 | 9% |

### 응답 코드 (normal 7건)

| 응답코드 | 수 | 의미 |
|----------|-----|------|
| 180 (Ringing) | 3 | 정상 수신, 벨 울림 |
| 603 (Decline) | 2 | UE가 통화 거절 |
| 400 (Bad Request) | 1 | 변이된 SIP 파싱 실패 |
| - | 1 | 기타 |

### 분석

- **seed 0~26**: timeout 다수 + 간헐적 normal. identity(무변이)도 timeout → 이전 세션 미정리 또는 SA 만료가 주 원인
- **seed 27~40**: 전부 unknown (0ms) → **UE가 등록 해제됨**. port resolve 실패로 즉시 unknown 반환
- **seed 23의 400 Bad Request**: wire/default 변이가 SIP 파서 경계를 건드림 — 추후 심화 분석 대상
- **CANCEL 효과**: seed 7(603 → 180), seed 21(603), seed 23(400 → 180) 등에서 이전 세션 정리 후 다음 케이스가 응답을 받는 패턴 관찰

### 주요 교훈

1. **SA 만료 감지 필요**: 연속 unknown이 나오면 UE 재등록 대기 후 재시도해야 함
2. **cooldown 부족**: 0.5초로는 CANCEL 처리 + 세션 정리에 불충분. 최소 2~3초 필요
3. **identity도 timeout**: 변이가 아닌 인프라 문제(세션 미정리)가 timeout의 주 원인
