# VoLTE Mutation Fuzzer — 구현 문서

> VoLTE/IMS 단말의 SIP 스택에 비정상 SIP 패킷을 전송하여 크래시, 스택 장애, 타임아웃 등을 자동 탐지하는 블랙박스 뮤테이션 퍼저
>

---

## 1. 프로젝트 개요

Fuzzer.md에서 정리한 Poet → Courier → Oracle 구조를 그대로 구현한다.

```
Generator ──→ Mutator ──→ Sender ──→ Oracle
 (정상 패킷)   (변이 적용)   (전송+수집)   (이상 판정)
                                          │
                                    Campaign
                                  (자동화 루프)
```

| 구성 요소 | 퍼저 이론상 역할 | 대응 모듈 |
|-----------|-----------------|-----------|
| **Poet** (입력 생성기) | 테스트 케이스 제작 | `generator` + `mutator` |
| **Courier** (전달자) | 대상에 케이스 전달 | `sender` |
| **Oracle** (판정기) | 이상 여부 판정 | `oracle` |

- Generator가 RFC 기반 정상 패킷을 만들고, Mutator가 여기에 결함을 주입하면 Poet 역할이 끝남
- Sender가 이 패킷을 대상에 보내고 응답을 수집하면 Courier
- Oracle이 응답 + 프로세스 상태 + 로그를 분석하여 이상 여부를 판정
- Campaign이 이 전체 흐름을 수백~수천 케이스 반복 실행하고 결과를 저장

## 2. 기술 스택

### 2.1 왜 이 조합인가

| 항목 | 선택 | 이유 |
|------|------|------|
| Python 3.14 | 언어 | 타입 힌트, `match` 문, StrEnum 등 최신 기능. 프로토콜 모델링에 적합 |
| [uv](https://github.com/astral-sh/uv) | 패키지 매니저 | pip 대비 10~100배 빠른 의존성 해석. Astral 제공 |
| [Pydantic](https://docs.pydantic.dev/) v2 | 데이터 검증 | SIP 패킷 구조를 Pydantic 모델로 정의하면 런타임 검증 + JSON 직렬화가 하나의 모델로 해결됨. 프로젝트 전 계층(패킷, 설정, 결과)에서 사용 |
| [Typer](https://typer.tiangolo.com/) | CLI | 타입 어노테이션 기반 인자 파싱. `fuzzer request INVITE` 같은 서브커맨드를 선언적으로 구성 가능 |

### 2.2 개발 도구

- **pytest** — 테스트. 현재 270개 테스트
- **Ruff** — 린터 + 포매터. Black, isort, flake8을 하나로 통합
- **ty** — 타입 체커
- **Poethepoet** — 태스크 러너 (`poe lint`, `poe format` 등)

### 2.3 외부 의존

- **Baresip** — 퍼징 대상 SIP 소프트폰. CLI 기반이라 자동 설치/프로비저닝이 가능
- **pgrep** — ProcessOracle에서 프로세스 생존 확인에 사용

### 2.4 설계 원칙

- **Pydantic 전면 적용**: SIP 메시지 하나를 Pydantic 모델로 정의하면 "이 패킷이 valid한가"를 자동으로 검증할 수 있음. 동시에 JSON으로 직렬화해서 파이프라인 사이에서 주고받기도 가능
- **동기 I/O**: 네트워크 퍼징은 RTT가 병목이라 비동기를 써도 큰 이점이 없음. 단순성 우선
- **Unix 파이프라인**: `generate | mutate | send` 로 각 단계를 독립 실행하거나 조합 가능. 디버깅 시 중간 결과를 파일로 빼서 확인할 수 있어 편함
- **contracts/core 분리**: 모듈마다 데이터 모델(contracts.py)과 로직(core.py)을 분리. 모듈 간 의존은 contracts만 참조해서 결합도를 낮춤

## 4. Generator — SIP 패킷 생성

Generator는 RFC 3261 및 확장 RFC를 따르는 **정상(valid) 패킷**을 만든다. 이 패킷이 뮤테이션의 시드 입력이 됨

### 4.1 SIP 카탈로그

`SIPCatalog`가 프로토콜 명세를 중앙 관리한다

- 14개 요청 메서드: ACK, BYE, CANCEL, INFO, INVITE, MESSAGE, NOTIFY, OPTIONS, PRACK, PUBLISH, REFER, REGISTER, SUBSCRIBE, UPDATE
- 75개 응답 코드: 1xx(Informational) ~ 6xx(Global Failure)
- 메서드마다 JSON 스키마, 전제조건(이 메서드를 보내려면 다이얼로그가 필요한가?), 필수/선택 헤더 정보를 포함

IANA SIP 레지스트리를 전수 조사해서 만든 것이므로 누락된 표준 메서드/응답코드는 없음

### 4.2 GeneratorSettings — 환경변수 기반 설정

SIP 헤더의 기본값을 환경변수(`VMF_GENERATOR_*`)에서 읽어온다

```
VMF_GENERATOR_VIA_HOST=proxy.example.com
VMF_GENERATOR_FROM_USER=remote
VMF_GENERATOR_TO_USER=ue
VMF_GENERATOR_REQUEST_URI_HOST=example.com
```

`.env` 파일이나 `os.environ`에서 읽으므로, Docker 환경이든 실험실 환경이든 설정 파일 하나만 바꾸면 됨

### 4.3 DialogContext — 다이얼로그 상태 추적

SIP는 stateful 프로토콜이므로 BYE, PRACK, UPDATE 같은 다이얼로그 내부 메서드를 생성하려면 상태가 필요함

- `call_id` + `local_tag` + `remote_tag` — 다이얼로그 식별 (이 세 값이 일치해야 같은 대화)
- `local_cseq`, `remote_cseq` — 트랜잭션 순서 보장
- `next_local_cseq()` — CSeq를 자동 증가
- `fork_for_reinvite()` — re-INVITE용 컨텍스트 복제

→ 현재는 stateless 퍼징만 구현. DialogContext는 향후 stateful dialog fuzzing에서 본격 활용 예정

### 4.4 생성 흐름

```
RequestSpec(method="INVITE")
  → SIPGenerator.generate_request(spec, context)
  → SIP_CATALOG에서 메서드별 모델 조회
  → Via, From, To, Contact, CSeq 등 헤더 빌드
  → 전제조건 검증 (다이얼로그 필요 여부)
  → 오버라이드 적용
  → SIPRequest (Pydantic 모델)
```

## 5. Mutator — 3계층 뮤테이션

Generator가 만든 정상 패킷에 의도적 결함을 주입하는 핵심 엔진

세 가지 레이어에서 각각 다른 수준의 변이를 적용함. Fuzzer.md 3.1에서 설명한 model/wire/byte 구분을 그대로 구현

### 5.1 Model 레이어 — 프로토콜 필드 단위

Pydantic 필드 값을 직접 조작. 프로토콜 의미를 이해하는 가장 정밀한 변이

| 대상 필드 | 오퍼레이터 | 노리는 것 |
|-----------|-----------|----------|
| `call_id` | replace_text | 다이얼로그 매칭 깨뜨리기 |
| `cseq.sequence` | replace_integer | 0, MAX_INT, 음수 — 시퀀스 처리 버그 |
| `max_forwards` | replace_integer | 0이면 전달 거부, 음수면 파서 오류 |
| `request_uri.host` | replace_host | 라우팅 로직 혼란 |
| `from_.parameters.tag` | replace_text | 발신자 태그 불일치 → 트랜잭션 매칭 실패 |
| `to.parameters.tag` | replace_text | 수신자 태그 불일치 |

- ex) "Via 헤더의 branch를 날리면 트랜잭션 매칭이 어떻게 되는가?"

### 5.2 Wire 레이어 — 헤더 구조 조작

Pydantic 모델을 와이어 텍스트로 렌더링한 뒤, 텍스트 수준에서 헤더 구조를 건드림

- **헤더 삭제**: Via나 CSeq 같은 필수 헤더를 날려버리면?
- **헤더 중복**: 같은 헤더를 여러 번 넣으면 파서가 어떻게 처리하는가?
- **헤더 순서 변경**: 순서에 의존하는 파서 구현 버그를 노림
- **Content-Length 변조**: 실제 바디보다 크거나 작게 → 버퍼 관련 버그

중간 표현으로 `EditableSIPMessage`를 사용. 이 표현은 **malformed 상태를 허용**함 — 정상적인 Pydantic 모델에서는 불가능한 구조(필수 헤더 없음, 중복 헤더 등)를 의도적으로 만들 수 있어야 하기 때문

### 5.3 Byte 레이어 — 원시 바이트 조작

와이어 텍스트를 바이트로 변환한 뒤 가장 무차별적인 방식으로 조작

- 비트 플립, 랜덤 바이트 삽입/삭제
- CRLF 구분자 변조 (`\r\n` → `\n`, `\r`, 또는 제거)
- NULL 바이트, 0xFF 등 경계값 주입

프로토콜 의미를 전혀 모르는 파서 레벨의 버그를 찾는 데 특화. 대부분의 패킷은 파싱 단계에서 걸러지겠지만, 걸러지지 않는 한 건이 크래시로 이어질 수 있음

### 5.4 뮤테이션 전략

| 전략 | 목적 | 적용 레이어 |
|------|------|------------|
| `default` | 범용 무작위 변이 | model, wire, byte |
| `state_breaker` | 다이얼로그/트랜잭션 상태 위반 유도 | model만 |

`state_breaker`는 `call_id`, `cseq.sequence`, `from_.parameters.tag`, `to.parameters.tag`, `request_uri.host` 다섯 개 필드를 집중 공격함 — 이 필드들이 SIP 다이얼로그/트랜잭션 상태를 식별하는 핵심이기 때문

wire/byte 레이어는 이미 충분히 파괴적이라 state_breaker를 별도로 구현할 필요가 없었음

### 5.5 재현성

모든 뮤테이션은 `seed` 값으로 결정론적 재현이 가능함. 같은 seed를 주면 같은 변이가 나옴

```bash
# seed=42로 항상 동일한 변이
uv run fuzzer mutate request INVITE --strategy state_breaker --seed 42
```

`MutatedCase`에 `records` 필드로 적용된 뮤테이션 이력이 기록됨:
```
MutationRecord(layer="model", operator="replace_text", target="call_id", before="abc", after="xyz")
```

→ 버그를 발견하면 seed만 기록해두면 언제든 동일 패킷을 재생산할 수 있음. 패치 후 회귀 테스트에도 사용 가능

## 6. Sender — 패킷 전달

뮤테이션된 패킷을 대상에게 보내고 SIP 응답을 수집하는 Courier 역할

### 6.1 전송 모드

| 모드 | 대상 | 설명 |
|------|------|------|
| `softphone` | 로컬 Baresip | 로컬에서 돌리는 소프트폰 대상. 완전 제어 가능 |
| `real-ue-direct` | 실제 UE | IMS 네트워크를 거치지 않고 UE에 직접 UDP 전송 |

softphone 모드가 기본. 실 단말 테스트 전에 소프트폰에서 먼저 검증하는 전략

### 6.2 프로토콜

- **UDP**: 기본. 한 요청에 대해 최대 8개 응답을 수집 (100 Trying → 180 Ringing → 200 OK 같은 다중 응답 처리)
- **TCP**: 전체 메시지 버퍼링, Content-Length 기반으로 메시지 경계 분리
- **TLS/SIPS**: 미구현

### 6.3 SendArtifact — 패킷 형태

뮤테이션 레이어에 따라 패킷 형태가 달라짐:

- model 레이어 변이 → `from_packet(SIPRequest)` — Pydantic 모델을 렌더링해서 전송
- wire 레이어 변이 → `from_wire_text(str)` — 변이된 텍스트를 그대로 전송
- byte 레이어 변이 → `from_packet_bytes(bytes)` — 바이트 배열 그대로 전송

한 단계씩 내려갈수록 원본에서 멀어지고, Sender는 받은 형태 그대로 소켓에 밀어넣음

### 6.4 응답 수집

`SendReceiveResult`에 전체 트랜잭션 결과가 담김:

- `outcome`: 최종 결과 분류 — provisional / success / error / timeout / send_error / invalid_response
- `responses`: 수신된 SIP 응답 목록
- `duration_ms`: 전송부터 최종 응답까지 걸린 시간
- `final_response`: 마지막 non-provisional 응답 (provisional만 온 경우 마지막 응답)
- `correlation_key`: Call-ID + CSeq 기반 요청-응답 매칭

→ 이 결과가 Oracle로 넘어감

## 7. Oracle — 이상 판정

패킷을 보낸 뒤 **대상에 무슨 일이 일어났는가**를 판정하는 모듈

Fuzzer.md 3.3에서 정리한 두 가지 축(소켓 응답 + 프로세스 상태)에 로그 패턴 매칭을 추가하여 세 축으로 구현함

### 7.1 Verdict 종류

| 판정(Verdict) | 의미 | 탐지 방법 |
|--------------|------|----------|
| `normal` | 정상 동작 | 아래 조건 모두 미해당 |
| `suspicious` | 비정상 응답 | 5xx 응답, 파싱 실패, 비정상 지연 |
| `timeout` | 응답 없음 (패킷 drop 또는 hang) | 소켓 타임아웃 |
| `crash` | 대상 프로세스 종료 | pgrep으로 프로세스 부재 확인 |
| `stack_failure` | 스택 트레이스 감지 | 로그 파일 패턴 매칭 |
| `unknown` | 판정 불가 (인프라 오류) | 전송 자체 실패 |

### 7.2 SocketOracle — 소켓 응답 기반 판정

`SendReceiveResult`를 받아서 소켓 수준의 판정을 내림

```
send_error        → unknown     (보내기 자체가 실패)
timeout           → timeout     (응답이 안 옴)
invalid_response  → suspicious  (응답이 왔는데 SIP로 파싱 불가)
status_code ≥ 500 → suspicious  (서버 에러)
elapsed > 임계값   → suspicious  (비정상적으로 느린 응답)
그 외             → normal
```

가장 기본적인 판정. 네트워크 퍼징이라 소켓 응답 분석이 1차 판단 기준이 됨

### 7.3 ProcessOracle — 프로세스 생존 확인

`pgrep -x <process_name>` 으로 대상 프로세스가 살아있는지 확인

- 프로세스가 없으면 → `crash` (confidence: 0.9)
- 프로세스가 있으면 → SocketOracle 결과에 `process_alive=True`만 추가

confidence가 1.0이 아닌 이유: pgrep 시점과 실제 crash 시점 사이에 갭이 있을 수 있고, 프로세스가 재시작됐을 가능성도 있음

### 7.4 LogOracle — 로그 파일 스택 트레이스 탐지

대상 프로세스(baresip 등)는 외부에서 실행되므로 stderr를 직접 캡처할 수 없음. 대신 **로그 파일을 incremental read**하여 스택 트레이스 패턴을 탐지

기본 탐지 패턴:

```
SIGSEGV                              # C 세그폴트
SIGABRT                              # 비정상 종료 시그널
Segmentation fault                   # 세그폴트 메시지
Assertion failed                     # 어서션 실패
core dumped                          # 코어 덤프
backtrace                            # 스택 백트레이스
Traceback (most recent call last)    # Python 예외
Exception in thread                  # Java 예외
\bFATAL\b                            # 치명적 오류
\bpanic:                             # Go panic
```

#### Incremental Read가 필요한 이유

캠페인은 수백 케이스를 연속 실행함. 매 케이스마다 로그 파일 전체를 다시 읽으면:
1. 이전 케이스에서 이미 탐지한 패턴을 다시 잡아서 false positive 발생
2. 로그가 커질수록 스캔 시간 증가

→ `after_position` 바이트부터 읽어서 새로 추가된 내용만 스캔. `(LogCheckResult, new_position)` 을 반환해서 호출자가 position을 추적

```
[로그 파일]
┌──────────────────────────────┐
│  기존 내용 (이미 스캔 완료)     │ ← after_position 이전, 건너뜀
├──────────────────────────────┤
│  새로 추가된 내용               │ ← 이 부분만 스캔
└──────────────────────────────┘
```

C 프로세스가 crash 난 뒤 로그에 바이너리 쓰레기가 섞일 수 있으므로 `errors="replace"`로 읽음

커스텀 패턴도 지원:
```python
LogOracle(patterns=("CUSTOM_PATTERN", r"my_app.*segfault"))
```

### 7.5 OracleEngine — 세 축 조합

세 가지 Oracle을 결합하여 최종 verdict를 산출

```
                    SendReceiveResult
                          │
                    SocketOracle
                     (소켓 판정)
                          │
                ┌─── log_path 있음? ──┐
                │                    │
           LogOracle             (스킵)
          (로그 스캔)                │
           │     │                  │
        매칭됨  미매칭               │
           │     │                  │
    stack_failure │                  │
                 ├──────────────────┘
                 │
          ┌── process_name 있음? ──┐
          │                       │
     ProcessOracle             (스킵)
       (생존 확인)                  │
       │       │                   │
    사망     생존                   │
      │       │                   │
   crash  socket 결과             │
              │                   │
              └───────────────────┘
                       │
                 최종 Verdict
```

판정 우선순위: **stack_failure > crash > socket verdict**

스택 트레이스는 crash의 **원인**을 담고 있음. 단순히 "프로세스가 죽었다"보다 "SIGSEGV로 죽었다"가 정보량이 많으므로 stack_failure를 우선

## 8. Campaign — 자동화 루프

수백~수천 개의 테스트 케이스를 사람 개입 없이 자동 실행하고 결과를 영속 저장하는 모듈

### 8.1 공격 범위 지정

캠페인 범위는 CLI 옵션으로 직접 지정한다.

- `--methods`: 요청 메서드 목록을 쉼표로 지정한다. 예: `OPTIONS,INVITE,MESSAGE,REGISTER`
- `--response-codes`: 응답 코드 목록을 쉼표로 지정한다. 예: `180,200,486`
- `--layer`: 뮤테이션 레이어를 지정한다. 예: `model,wire,byte`
- `--strategy`: 뮤테이션 전략을 지정한다. 예: `default,state_breaker`

메서드와 응답 코드는 함께 지정할 수 있고, 둘 다 비워 두면 전체 SIP 요청 메서드 집합이 기본값으로 사용된다.

### 8.2 테스트 케이스 생성

`CaseGenerator`가 설정값에서 `method × layer × strategy` 조합을 생성

```python
for method in config.methods:
    for layer in config.layers:
        for strategy in config.strategies:
            yield CaseSpec(case_id, seed, method, layer, strategy)
```

layer별로 지원하는 전략이 다름 — wire/byte는 `state_breaker`를 지원하지 않으므로 유효하지 않은 조합은 자동으로 제외됨

### 8.3 실행 흐름

```
for each CaseSpec:
    1. Generator로 정상 패킷 생성
    2. Mutator로 변이 적용 (spec의 layer/strategy/seed)
    3. Sender로 전송 + 응답 수집
    4. Oracle로 판정 (소켓 + 로그 + 프로세스)
    5. ResultStore에 결과 기록
    6. crash/stack_failure면 stderr에 재현 명령어 출력
    7. cooldown 대기 (기본 0.2초)
```

cooldown이 필요한 이유는 Fuzzer.md 4.3에서 설명 — 연속 패킷 전송 시 UDP 드롭이나 서버 과부하로 false positive가 생길 수 있음

### 8.4 결과 저장 — JSONL

결과는 **append-only JSONL** 형식으로 저장. 캠페인 도중 크래시가 나도 이미 기록된 결과는 보존됨

```
campaign.jsonl
├── {"type":"header", "campaign_id":"a1b2c3", "status":"running", "config":{...}}
├── {"type":"case", "case_id":0, "method":"INVITE", "verdict":"normal", ...}
├── {"type":"case", "case_id":1, "method":"OPTIONS", "verdict":"suspicious", ...}
├── {"type":"case", "case_id":2, "method":"INVITE", "verdict":"crash", ...}
└── {"type":"footer", "campaign_id":"a1b2c3", "status":"completed", "summary":{...}}
```

- header: 캠페인 시작 시 기록 (설정, 시작 시각)
- case: 각 테스트 케이스 결과를 즉시 append
- footer: 캠페인 종료 시 요약 통계 추가

SQLite 대신 JSONL을 선택한 이유: 스키마 없이 바로 쓸 수 있고, `jq` 같은 도구로 분석할 수 있으며, append-only 특성이 crash-safe에 유리

### 8.5 재현 명령

각 케이스에 `reproduction_cmd`가 자동 생성됨:

```bash
uv run fuzzer mutate request INVITE \
  --strategy state_breaker --layer model --seed 42 \
  | uv run fuzzer send packet --target-host 10.20.20.2 --target-port 5060
```

seed + strategy + layer 조합이 결정론적이므로 이 명령을 다시 돌리면 동일한 뮤테이션 패킷이 나옴

## 9. CLI 사용법

### 9.1 단일 패킷 수동 테스트

Unix 파이프라인으로 각 단계를 조합

```bash
# 패킷 생성만
uv run fuzzer request INVITE

# 생성 + 뮤테이션
uv run fuzzer request OPTIONS \
  | uv run fuzzer mutate packet --layer wire --seed 42

# 전체 파이프라인: 생성 → 뮤테이션 → 전송
uv run fuzzer request INVITE \
  | uv run fuzzer mutate packet --strategy state_breaker --seed 42 \
  | uv run fuzzer send packet --target-host 10.20.20.2 --target-port 5060
```

각 단계의 출력이 JSON이므로 중간에 파일로 빼서 확인하거나 `jq`로 필터링할 수 있음

### 9.2 캠페인 실행

```bash
uv run fuzzer campaign run \
  --target-host 10.20.20.2 \
  --target-port 5060 \
  --methods OPTIONS,INVITE,MESSAGE,REGISTER \
  --layer model,wire,byte \
  --strategy default,state_breaker \
  --max-cases 1000 \
  --timeout 5 \
  --cooldown 0.2 \
  --process-name baresip \
  --log-path /var/log/baresip.log \
  --output results/campaign.jsonl
```

| 옵션 | 기본값 | 설명 |
|------|--------|------|
| `--methods` | 전체 SIP 요청 메서드 | 퍼징할 요청 메서드 목록. 쉼표 구분으로 여러 개 지정 가능 |
| `--response-codes` | (없음) | 퍼징할 SIP 응답 코드 목록. 쉼표 구분으로 여러 개 지정 가능 |
| `--strategy` | default,state_breaker | 뮤테이션 전략. 쉼표 구분으로 여러 개 지정 가능 |
| `--layer` | model,wire,byte | 뮤테이션 레이어 |
| `--max-cases` | 1000 | 최대 테스트 케이스 수 |
| `--timeout` | 5.0 | 소켓 타임아웃(초) |
| `--cooldown` | 0.2 | 케이스 간 대기(초) |
| `--process-name` | baresip | 프로세스 생존 확인 대상 |
| `--no-process-check` | false | 프로세스 확인 비활성화 |
| `--log-path` | (없음) | 로그 파일 경로. 지정하면 LogOracle 활성화 |
| `--transport` | UDP | 전송 프로토콜 |
| `--mode` | softphone | 전송 모드 |

### 9.3 결과 분석 및 재현

```bash
# 전체 결과 요약
uv run fuzzer campaign report results/campaign.jsonl

# 이상 케이스만 필터
uv run fuzzer campaign report results/campaign.jsonl \
  --filter suspicious,crash,stack_failure

# 특정 케이스 재실행 (동일 패킷을 다시 보냄)
uv run fuzzer campaign replay results/campaign.jsonl --case-id 42
```

## 10. 구현 현황

### 10.1 완료

| Phase | 모듈 | 내용 |
|-------|------|------|
| Phase 1 | `sip/` | SIP 메시지 분류 + 카탈로그 (14 메서드, 75 응답코드) |
| Phase 2 | `generator/` | RFC 기반 정상 패킷 생성, 환경변수 설정, DialogContext |
| Phase 3 | `mutator/` | 3계층 뮤테이션(model/wire/byte), 2가지 전략(default/state_breaker), seed 재현 |
| Phase 4 | `sender/` | UDP/TCP 전송, 멀티응답 수집, real-ue-direct 모드 |
| Phase 5 | `oracle/` | SocketOracle + ProcessOracle + LogOracle 3축 판정 |
| Phase 5 | `campaign/` | 직접 지정한 method/layer/strategy 조합 실행, JSONL 저장, report/replay CLI |

전체 테스트: **270개** 통과

### 10.2 미구현

| 기능 | 설명 |
|------|------|
| Stateful dialog fuzzing | INVITE→PRACK→BYE 시퀀스 내부에서 뮤테이션. DialogContext 기반은 준비되어 있으나 캠페인 루프가 stateless 전용 |
| ADB logcat 연동 | Android 실 단말의 crash/ANR 탐지. 현재 LogOracle은 파일 기반이라 logcat 스트림 대응 필요 |
| TLS/SIPS 전송 | 암호화 전송 |
| 병렬/분산 퍼징 | 다수 단말 동시 퍼징. 현재는 동기 순차 실행 |
